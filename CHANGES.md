# CHANGES — code-review

Ветка содержит результаты полного code review и оптимизации проекта.

## Новые файлы

### `pipeline.py`
Общая инициализация search pipeline, вынесенная из `main.py` и `api.py`.
- Функция `build_search_pipeline()` — единая точка инициализации для CLI и API
- Функция `load_or_build_chunks()` — загрузка чанков из кэша или парсинг PDF
- Все пути через `Path(__file__).parent` — запуск из любой директории
- Тайминги холодного старта по каждому из 4 этапов + суммарное время

### `CLAUDE.md`
Документация проекта: архитектура, параметры, структура файлов, инструкции по запуску, технические детали, лог исправлений.

## Изменённые файлы

### `api.py`
- Убрано дублирование инициализации — теперь использует `pipeline.py`
- Добавлен `CORSMiddleware` для локальной разработки через VS Code Go Live
- Добавлен **LRU-кэш запросов** (64 записи, `OrderedDict` + `threading.Lock`) — повторный запрос обслуживается мгновенно
- Добавлены **тайминги** каждого этапа поиска в лог: `embed=Xms bm25=Xms rrf=Xms rerank=Xms total=Xms`

### `main.py`
- Убрано дублирование инициализации — теперь использует `pipeline.py`
- Все пути и гиперпараметры импортируются из `pipeline.py`

### `parsing.py`
- Удалены неиспользуемые функции: `pdf_to_plain_text`, `save_plain_text`, `load_plain_text`
- **Расширено удаление псевдокода** в `_remove_algo_blocks()` — переписан как конечный автомат:

| Тип блока | Пример | Как детектируется |
|---|---|---|
| while/for/if блоки | `while x do ... end while` | `_ALGO_BLOCK_START_RE` / `_ALGO_BLOCK_END_RE` |
| Pascal begin/end | `begin ... end` | `_PASCAL_BEGIN_RE` / `_PASCAL_END_RE` |
| Заголовок алгоритма (рус.) | `Алгоритм 2.1:` | `_ALGO_HEADER_RU_RE` |
| Блок кода без кириллицы | ≥4 строки с `:=`, `->`, `return`… | `_CODE_SIGNS_RE` + буфер `noncyrillic_run` |

### `embedding.py`
- Удалён дубликат функции `load_chunks` (единственная реализация — `utils/load_chunks.py`)
- Переименован `faiss_id` → `array_idx` (FAISS не используется, применяется numpy)

### `keyword_search.py`
- `MorphAnalyzer` переведён на **ленивую инициализацию** (`_get_morph()`) — не создаётся при импорте, только при первом вызове BM25

### `trainer_mode.py`
Полная переработка генератора вопросов. До исправлений генерировалось ~2 вопроса на весь документ; после — десятки.

**Причины проблемы и исправления:**

| # | Проблема | Исправление |
|---|---|---|
| 1 | `clean_text()` переводил текст в нижний регистр перед natasha — NLP качество деградировало | Разделено: `clean_text_for_nlp()` (регистр сохранён) и `clean_text_for_check()` (lowercase только для сравнения ответов) |
| 2 | Типы `property`, `comparison`, `cause` не имели шаблонов вопросов — предложения классифицировались, но тихо выбрасывались | Добавлены шаблоны для всех 7 типов |

| 4 | Лемма `"как"` в dependency-паттерне давала ложные срабатывания почти на каждое предложение | Удалена из списка лемм |
| 5 | `len(answer.split()) > 12` обрезал допустимые ответы | Лимит поднят до 20–22 слов |
| 6 | Порог `min_score=0.65` был избыточно высоким | Снижен до `0.55` |
| 7 | Отсутствие диагностики — невозможно определить, на каком шаге отсеиваются предложения | Добавлен подробный вывод: `sents= short= long= no_type= no_pattern= low_score= ok= types=` |
| 8 | Удалена неиспользуемая константа `INLINE_HEADER_RE` | — |
| 9 | Голые `except:` | Заменены на `except Exception:` |

Расширены шаблоны определений: было 4, стало 12 (`"X определяется как Y"`, `"Под X понимают Y"`, `"термином X называют Y"` и др.)

## Замеры производительности (CPU, Apple Silicon, без GPU)

### Cold start

| Этап | Время |
|---|---|
| Загрузка чанков из кэша | ~10 ms |
| Загрузка моделей (e5-large + bge-reranker) | ~13 000 ms |
| Загрузка эмбеддингов (numpy) | ~5 ms |
| Загрузка BM25 | ~50 ms |
| **Итого** | **~14 000 ms** |

### Один поисковый запрос

| Запрос | Embedding | BM25 | RRF | Reranker | Итого |
|---|---:|---:|---:|---:|---:|
| что такое граф | 5215 ms | 83 ms | <1 ms | 7486 ms | **12 784 ms** |
| алгоритм обхода в ширину | 7752 ms | 5 ms | <1 ms | 5426 ms | **13 182 ms** |
| дерево поиска | 7317 ms | 1 ms | <1 ms | 5590 ms | **12 909 ms** |
| сортировка слиянием | 6916 ms | 6 ms | <1 ms | 7795 ms | **14 716 ms** |
| динамическое программирование | 7131 ms | 7 ms | <1 ms | 5732 ms | **12 871 ms** |

Bottleneck — оба нейросетевых этапа работают на CPU. При наличии GPU (CUDA/MPS) итоговое время сократится до ~200–500 ms.
Повторные идентичные запросы обслуживаются из LRU-кэша мгновенно.
